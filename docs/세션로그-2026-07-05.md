## §0. 아키텍처 대전환 — 방법 B (웹 관제 + 로컬 에이전트)

**배경:** 클라우드(Railway, 데이터센터 IP)에서 네이버 스크래핑/자동화가 차단·502 발생. 무거운 일은 각 사용자 PC로 분산하는 구조로 전환.

**설계:**
```
[웹(Vercel)] → [클라우드 API(Railway)]  … 작업을 Postgres(agent_jobs)에 적재만
                    ▲ 폴링                ▼
        [로컬 에이전트(사용자 PC)] ── GET /api/agent/next-job (아웃바운드 폴링)
           → 집 IP로 Playwright 실행 → POST /api/agent/job-result
[웹] ── GET /api/agent/jobs/{id} 폴링 → 결과 표시
```
- **스위치:** `EXECUTION_MODE=cloud` 면 job 적재(에이전트 위임), `local`(설치형)이면 인프로세스 즉시 실행. **한 코드로 두 배포 지원.**
- 신규: `agent_jobs` 테이블, `backend/jobs.py`(enqueue/claim/complete/get + PERSISTERS 훅), `backend/routers/agent_router.py`, **`agent.py`**(폴러).
- 커밋: `1a7451a`(P1 골격+SEO검색), `3c9ede9`(P2 analyze), `27a2c4c`(분석 일괄化+DB동기화 훅).

**에이전트 처리 대상(job type):** seo_search, seo_analyze, seo_cafe_urls, seo_top3, seo_cafe_post, blog_index, place_analyze, place_fetch_mid, place_fetch_reviews(되돌림), **register_account(기기인증)**, **auto_post(발행)**.

## §1. 클라우드 DB 동기화 (플레이스)

- 휘발성 `data/ranking.db`(SQLite) → **Postgres 이관**: 모델 `PlaceTracked`, `PlaceRankSnapshot`.
- `persist_place_snapshot` + `jobs.PERSISTERS` 훅 — 에이전트가 반환한 순위 결과를 클라우드 Postgres에 일자별 upsert → 웹의 관심목록·히스토리가 유지됨.
- `/track /tracked /history` 를 유저별 Postgres 조회로 재작성 + 관심목록 삭제(`/untrack`) 추가.
- 커밋: `b10432f`, `2a4fe00`.

## §2. 백엔드 다운 사고 → 복구 (크래시 루프 → OOM)

- **증상:** 잦은 재배포 후 백엔드가 `Waiting for application startup`에서 5초마다 재시작(크래시 루프), 이후 실행 중 OOM.
- **원인 1(startup 블로킹):** lifespan의 `scheduler_service.start()` — 스케줄 잡은 전부 스크래핑이라 클라우드 불가. → **cloud 모드에선 스케줄러 미기동**(`b728167`).
- **원인 2(OOM):** shopping_router/keyword_seo/golden_keyword가 시작 시 **Kiwi 형태소 모델 3개 로드 = ~850MB** → 트라이얼 RAM 초과. → **cloud 모드에선 Kiwi 미로드**(`5ab8c3a`). **실측: 앱 RAM 1062MB → 209MB(80%↓)**. 크래시 루프 완전 해소.

## §3. 플레이스 소식(새소식 원고) 수정
